# What is Jest and Testing Types

Software testing ensures application code correctness. Testing is divided into unit tests (verifying isolated elements), integration tests (verifying components interaction), and End-to-End tests (simulating full user flows).

**Core Concepts:**
* **Unit Testing:** Focuses on testing single functions, components, or modules in isolation.
* **Integration Testing:** Checks that multiple modules, database models, or external services function correctly together.
* **Test Runner:** Jest is a comprehensive JavaScript test runner providing testing environments, assertions, mocking tools, and code coverage metrics.

```javascript
// Simple Jest Unit Test
const sum = (a, b) => a + b;

test('adds 1 + 2 to equal 3', () => {
  expect(sum(1, 2)).toBe(3);
});
```

# React Testing Library Philosophy and DOM Queries

React Testing Library (RTL) is the modern standard for validating React components. It encourages testing component behavior similar to real user interactions rather than testing raw implementation configurations.

**Core Concepts:**
* **User-Centric Philosophy:** Tests query rendered DOM tree elements rather than asserting internal state variables.
* **Accessible Selectors:** RTL queries elements by accessible properties like roles (`screen.getByRole`) or labels (`screen.getByLabelText`).
* **Visual Stability:** Tests are less brittle because refactoring internal code doesn't break query assertions if the UI layout is unchanged.

```javascript
import React from 'react';
import { render, screen } from '@testing-library/react';

function SimpleButton({ label }) {
  return <button className="btn-primary">{label}</button>;
}

// RTL verification test
test('renders Button component with correct label', () => {
  render(<SimpleButton label="Click Me" />);
  const buttonElement = screen.getByRole('button', { name: /click me/i });
  expect(buttonElement).toBeInTheDocument();
});
```

# Mocking External Services and API Requests

Live dependencies (APIs, files, external modules) are replaced with mock configurations during testing. Mocking keeps test executions fast, offline, and deterministic.

**Core Concepts:**
* **Jest Spy Spies:** Tracking function execution, checking call counts, and inspecting passed arguments.
* **Module Mocking:** Replacing complete external libraries (like `axios` or database drivers) with custom mocked objects.
* **Deterministic Runs:** Eliminates network dependence, allowing tests to run locally and test error conditions like request timeouts.

```javascript
import React from 'react';
import { render, screen, fireEvent } from '@testing-library/react';

function ActionButton({ onClick }) {
  return <button onClick={onClick}>Submit Action</button>;
}

test('calls onClick callback when clicked', () => {
  const mockCallback = jest.fn(); // Create mock function
  render(<ActionButton onClick={mockCallback} />);
  
  const button = screen.getByText(/submit action/i);
  fireEvent.click(button);
  
  expect(mockCallback).toHaveBeenCalledTimes(1); // Assert call count
});
```

# Custom DOM Assertions and Expectations

Assertions form the core validation checks in test scripts, verifying application outputs against expected criteria. RTL extends Jest's default assertions with custom matchers.

**Core Concepts:**
* **Expect Matchers:** Default Jest expectations like `toBe()`, `toEqual()`, `toBeNull()`, and `toBeUndefined()`.
* **RTL DOM Matchers:** Custom Jest matchers like `toBeInTheDocument()`, `toBeVisible()`, `toBeDisabled()`, and `toHaveClass()`.
* **State Verification:** Ensures user interactions transition component layouts correctly (e.g. clicking "Open menu" displays dynamic divs).

```javascript
import React, { useState } from 'react';
import { render, screen, fireEvent } from '@testing-library/react';

function ToggleMessage() {
  const [show, setShow] = useState(false);
  return (
    <div>
      <button onClick={() => setShow(!show)}>Toggle</button>
      {show && <p data-testid="msg">Secret Message</p>}
    </div>
  );
}

test('shows secret message on toggle click', () => {
  render(<ToggleMessage />);
  
  // Verify element is not present initially
  expect(screen.queryByTestId('msg')).toBeNull();
  
  fireEvent.click(screen.getByText(/toggle/i));
  
  // Assert element is now present and visible
  expect(screen.getByTestId('msg')).toBeInTheDocument();
});
```

# Measuring Code Coverage

Code coverage represents the percentage of code statements, branch logic, and functions executed during test runs. Coverage metrics help identify untested execution paths.

**Core Concepts:**
* **Statements Coverage:** Verifies if every line of code was executed during tests.
* **Branches Coverage:** Verifies if every decision fork (e.g., both `if` and `else` blocks) was checked.
* **CI Integration:** Setting baseline thresholds (e.g. 80% coverage) to block code merges that drop software quality standards.

