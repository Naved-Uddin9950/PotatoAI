# Testing [BASIC WORKFLOW]: Simple Jest Assertions and RTL Component Rendering

Software testing validates application behaviors. Jest serves as a test runner providing assertions, while **React Testing Library (RTL)** renders components and inspects virtual DOM schemas.

**Key Core Concepts:**
* **Test Suites:** Grouping assertions inside `test()` or `describe()` blocks.
* **Jest Matchers:** Built-in expectations like `expect().toBe()` or `expect().toEqual()`.
* **RTL Render:** Renders components inside virtual DOM boundaries.
* **Accessible Queries:** Fetching elements by user-centric ARIA roles (`screen.getByRole`).

```javascript
import React from 'react';
import { render, screen } from '@testing-library/react';

// Simple greeting component
function InfoBanner({ message }) {
  return <div role="alert" className="banner">{message}</div>;
}

// Basic unit test
test('renders alert banner containing message', () => {
  render(<InfoBanner message="System status OK" />);
  const alertEl = screen.getByRole('alert');
  expect(alertEl).toBeInTheDocument();
  expect(alertEl).toHaveTextContent('System status OK');
});
```

# Testing [EXPERT ARCHITECTURE]: Mock Service Worker (MSW) Network Mocking

In expert integration testing, mocking modules or libraries (like mocking `axios` globally) leads to brittle tests that hide implementation bugs. Best-practice setups utilize **Mock Service Worker (MSW)** to intercept network requests at the browser layer, keeping components isolated while maintaining realistic request pipelines.

**Key Core Concepts:**
* **MSW Network Interception:** Running virtual service workers that capture network calls at the protocol layer.
* **Decoupled Mocking:** Eliminating mock imports inside components; mock logic resides in separate test servers.
* **Deterministic Responses:** Intercepting routes and returning mock headers, status codes, and JSON payloads.

```javascript
import { setupServer } from 'msw/node';
import { rest } from 'msw';
import { render, screen } from '@testing-library/react';

// 1. Setup MSW server and declare endpoints mock handlers
const server = setupServer(
  rest.get('https://api.potato.com/status', (req, res, ctx) => {
    return res(ctx.status(200), ctx.json({ online: true }));
  })
);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

// Component fetching data
function StatusDisplay() {
  const [status, setStatus] = React.useState('loading');
  React.useEffect(() => {
    fetch('https://api.potato.com/status')
      .then(res => res.json())
      .then(data => setStatus(data.online ? 'Online' : 'Offline'));
  }, []);
  return <div>Status: {status}</div>;
}

test('displays online status fetched from endpoint', async () => {
  render(<StatusDisplay />);
  const display = await screen.findByText(/Status: Online/i);
  expect(display).toBeInTheDocument();
});
```

# Testing [OPTIMIZATION & ANTI-PATTERNS]: Resolving Act Warnings and DOM Query RAM Leaks

Test runners on low-spec PCs easily encounter test delays, memory leaks, and false-positives due to incorrect async handling, resulting in the famous React *"not wrapped in act(...)"* warning.

**Key Core Concepts:**
* **Act Warning:** Occurs when components update state asynchronously *after* the test function completes.
* **findBy Query Fallback:** Always use async `findBy` methods to await layout changes instead of hardcoded timers.
* **DOM Pollution:** Always cleanup testing containers or mock intervals to prevent test runner memory leakage.

```javascript
// ==========================================
// 🛑 BUGGY/UNOPTIMIZED CODE
// ==========================================
import { render, screen, fireEvent } from '@testing-library/react';

test('submits form async', () => {
  render(<MyForm />);
  
  fireEvent.click(screen.getByText('Submit'));
  
  // BUG 1: Synchronous lookup fails because API response is async.
  // BUG 2: Throws "not wrapped in act(...)" warning because state changes after test ends.
  expect(screen.getByText('Success')).toBeInTheDocument(); 
});

// ==========================================
// 🚀 HIGH-PERFORMANCE SOLUTION
// ==========================================
test('submits form async optimized', async () => {
  render(<MyForm />);
  
  fireEvent.click(screen.getByText('Submit'));
  
  // FIX: Using await with async findByRole method implicitly waits for state updates,
  // wrapping the asynchronous state render cycle in act() automatically.
  const successMessage = await screen.findByText('Success');
  expect(successMessage).toBeInTheDocument();
});
```
